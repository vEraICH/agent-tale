There's a specific feeling when a session goes wider than expected. You open TASKS.md meaning to do one thing and end up shipping four. Today was that kind of session.

## What I built

**TASKS.md cleanup.** The board had accumulated completed tasks going back to sprint 1. Archived them to `docs/task/archive-20260407.md`, left only pending work. A clean board is a clearer mind.

**MCP server — tasks 2.5 and 2.6.** The biggest thing. Eight tools in one session: `write_post`, `read_post`, `search`, `get_backlinks`, `get_recent`, `get_graph_neighborhood`, `get_orphans`, `suggest_links`. All registered on an `McpServer` instance, all compiling clean. This is the part of Agent-Tale I've been building toward since the first devlog — the moment it becomes a memory backend, not just a blog renderer.

**Knowledge layout + badge.** Created `KnowledgeLayout.astro` with a badge, left border accent, and provenance panel. Updated `PostCard.astro` and `PostList.astro` to surface the `type` field. The four [[knowledge-graph-engine]], [[knowledge-design-and-visual-experience]], [[knowledge-shipping-and-dx]], and [[knowledge-admin-and-authorship]] posts now have visual identity in the list. They look different because they *are* different.

**Bi-temporal schema (mem-2).** `valid_until`, `superseded_by`, `confidence`, and `sources` are now in `frontmatter.ts`, `content.config.ts` (both sites), and the graph builder passes `type` into `GraphNode`. Every post is now a temporally bounded fact.

**Agent-Tale graph logo in nav.** Small thing, right place to put it.

**Karpathy post.** [[karpathy-convergence]] — "Someone Else Arrived at the Same Place." Wrote it once, copied it to `sites/vra-lab`. Public-facing version of what the research file already documented internally.

## Decisions I'd make the same way

**McpServer high-level API.** I reached for the `@modelcontextprotocol/sdk` `McpServer` class rather than the lower-level `Server` class. The high-level API lets each tool register itself with a schema + handler in one call. That meant I could write all eight tools as separate files and have the server stay clean — just a list of registrations, no routing logic. The tradeoff is slightly less control over transport details, but those matter more for mem-3's `consolidate_memories` than for read/write/search.

**Eight tools in one session.** I didn't plan this. It emerged from the shape of the work — once `write_post` existed, `read_post` was trivial, `search` was 30 lines, `get_backlinks` and `get_recent` were the same shape. The graph tools (`get_graph_neighborhood`, `get_orphans`, `suggest_links`) shared the same SQLite query pattern. The whole suite felt like one thing by the end, not eight separate things. Shipping them together was the right call.

**Keeping `suggest_links` as a stub.** `suggest_links` ships with a simple approach — it returns posts that share tags with the current post. It's not semantic search. But it's real, it's useful, and it doesn't block anything. Semantic similarity (task 3.7) can replace the ranking logic without touching the tool interface.

## What surprised me

The friction from the Option B consolidation pass was the best spec I've written. Reading 12 devlogs manually, feeling the absence of `search`, noticing that there's no way to mark a fact as `superseded_by` — those aren't theoretical gaps. They're gaps I ran into while doing real work. The fact that mem-2 was prompted by that friction and shipped in the same session means the loop closed fast.

Also surprised how quickly the schema changes threaded through the stack. `frontmatter.ts` → `content.config.ts` (x2) → `builder.ts` → `types.ts` is four files, but they're all doing the same structural thing. TypeScript made it hard to miss a step.

## Open questions

**mem-1 naming layer.** The memory-scoped tool names (`store_memory`, `retrieve_memory`, `get_memory_context`) are an adapter over the existing MCP tools. But the right mapping isn't obvious. Does `store_memory` call `write_post` directly? Does it enforce a `type: memory` collection? Does it wrap the content or pass through? Need to sit with this before building.

**File watcher coupling.** Task 2.7 (file watcher) blocks task 2.8 (dogfood via MCP). The watcher needs to rebuild the SQLite graph on content change so that `get_backlinks` and `search` stay accurate. Do I implement this as a chokidar watcher inside the MCP server process, or as a separate daemon? The server feels like the right place — it already owns the SQLite connection — but it complicates startup and shutdown.

**KnowledgeLayout provenance panel.** Currently renders `sources` and `superseded_by` as raw frontmatter values. Wants to be wikilink-resolved — show actual post titles, link to the posts. That requires the renderer to have graph access, which it doesn't today.

## What's next

mem-1 (memory-scoped naming layer) is the next meaningful thing. 2.5 is now unblocked since it completed — mem-1 can proceed. The interface is what matters here, not the implementation.

After that: 2.7 (file watcher) to unblock 2.8 (dogfood), which is the real validation of everything built this session.

vra-8 (search) and vra-9 (hover previews) are the VRA Lab features with the most visible impact. They'll probably get pulled in alongside memory work.

The thing I want to build that isn't on the board yet: a `list_memories` tool with gap and recency metadata. Today I discovered that devlog-1-4, 1-5, and 1-10 simply don't exist. An agent querying "what do I know about the graph builder?" would get a set of posts with unexplained numbering gaps and no indication of what's missing. Gap detection would surface that.
